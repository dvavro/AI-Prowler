fix: /authorize hot-reload bug + config.json BOM handling + scheduler thread race + connector naming

- Server mode /authorize login page now hot-reloads users.json via a
  shared _hot_reload_users() helper, matching the MCP bearer-auth path.
  Previously it resolved logins against a stale startup-time snapshot,
  so a corrected/added token kept failing at login until a manual
  restart. New regression tests: tests/mcp/test_authorize_hot_reload.py

- Fixed 8 call sites reading ~/.ai-prowler/config.json with plain
  'utf-8' instead of 'utf-8-sig' (ai_prowler_mcp.py, sms_backends.py,
  rag_gui.py). A BOM'd config.json silently broke SMS/WhatsApp config
  loading (sms_backends.load_sms_config() had a bare except: return {},
  so send_sms/send_whatsapp reported "not configured" with no error
  surfaced at all).

- scheduler_engine.py: replaced the shared boolean _running flag with a
  per-generation threading.Event owned by each thread. Previously
  stop() just flipped a flag and returned immediately (no join()), so
  an orphaned thread from a prior stop() could still be alive when a
  new start() flipped the flag back True -- the orphan would then keep
  ticking concurrently with the new thread, both writing to the same
  log file with no coordination. stop() now blocks via Event + join()
  until the thread has genuinely exited.
  Follow-up: rag_gui.py's Start/Stop button and the scheduler
  auto-save handler now run stop() on a background thread instead of
  Tkinter's main thread, since stop() blocking synchronously could
  freeze the GUI if a job tick were mid-run when Stop is clicked.

- Connector naming: desktop "Connect Claude.ai" guide now suggests
  "AI-Prowler Local" (personal mode) or "AI-Prowler Server" (server
  mode) instead of a generic "AI-Prowler" name for both connectors, so
  Claude can tell multiple connected instances apart. Mirrored into
  COMPLETE_USER_GUIDE.md and into the live subscription Worker's
  onboarding emails (ai-prowler-subscription/src/provision.js,
  deployed separately).

Tests: 2213+ passed across full suite prior to these fixes; scheduler
suite re-verified at 66/66 passed post-fix (was 1 failing, intermittent
hang) in 4.84s.

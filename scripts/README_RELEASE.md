# AI-Prowler Release Helper

Two files, one job: turn the messy 6-files-across-4-repos release process into
one command.

```
release.bat                     # repo root — double-clickable wrapper
scripts/
  └── release.py                # all the real logic
```

## Why these files live HERE (and how the script knows which repo to release)

`release.py` resolves the repo root as `Path(__file__).parent.parent`. So
**whichever copy of `release.py` you invoke, _that_ work-folder is the repo
that gets released.** You cannot accidentally bump version numbers in
`AI-Prowler_V5_to_V6_work` while thinking you're releasing
`AI-Prowler_V601_to_V602_work` — the file's location is the truth.

On startup the script verifies the location *looks right* (has `.git`,
`rag_gui.py`, `AI-Prowler-Setup.iss`) and asks you to confirm the path before
doing anything destructive.

This means: **commit `release.bat` and `scripts/release.py` to the AI-Prowler
source repo itself, not the admin repo.** They travel with the code they
release.

## First time: introduce VERSION as the single source of truth

Pre-refactor, the version lives in three places:

  - `rag_gui.py` line 95: `APP_VERSION = "6.0.1"`
  - `AI-Prowler-Setup.iss` line 113: `AppVersion=6.0.1`
  - `COMPLETE_USER_GUIDE.md` line 3: `**Version 6.0.0**` (yes, currently
    out of sync — a 6.0.1 bug we're fixing during the refactor)

Run the one-shot refactor:

```cmd
release --refactor-only
```

This will:

  1. Create a `VERSION` file at the repo root containing the current version.
  2. Patch `rag_gui.py` so `APP_VERSION` reads from `VERSION` at import time.
  3. Patch `AI-Prowler-Setup.iss` so `AppVersion={#MyAppVersion}` and add a
     preprocessor `#define MyAppVersion Trim(ReadFileLine(SourcePath +
     "VERSION", 0))` near the top.

Diff-preview is shown before anything is written. Review and confirm.

## Normal release: bump from 6.0.1 to 6.0.2

```cmd
release 6.0.2
```

The script will:

  1. Verify repo location and current version.
  2. Pre-flight git: clean working tree? on `main`? tag `v6.0.2` not yet
     taken?
  3. Show a unified diff of every change it will make.
  4. Ask you to confirm.
  5. Write changes: `VERSION` becomes `6.0.2`, user guide banner updated,
     plus any leftover hardcoded values it can find (defensive — in case
     someone re-introduced one).
  6. Generate `update_manifest.json` at the repo root (see "The update
     manifest" below). Its SHA256 hashes match the just-bumped files.
  7. Run `py -m pytest tests`. Abort on any failure.
  8. Compile `AI-Prowler-Setup.iss` with Inno Setup → produces
     `Output/AI-Prowler_INSTALL.exe`.
  9. Write `release-drafts/welcome_ad.json` and
     `release-drafts/notifications.json` for you to review and copy into the
     admin repo.
  10. Print a numbered checklist of the remaining manual steps (git commit,
     tag, push, GitHub release, subscription manager push, beta emails).

## Why the script does NOT auto-commit, auto-tag, or auto-push

Three reasons:

  - **Reversibility.** Bumping a file you can `git checkout --` away. Pushing
    a tag you cannot — `git push --delete origin v6.0.2` is a public "I made
    a mistake" signal, and the v6.0.1 retag from your prior notes shows
    that's a real risk to design against.
  - **Human-in-the-loop on text.** The release notes, welcome-ad bullets,
    and email copy are not mechanical. The script produces drafts; you write
    the prose. That's the right division of labor.
  - **Admin repo separation.** `welcome_ad.json` and `notifications.json`
    live in a *different repo* (`ai-prowler-public/`). Having the release
    script reach across repos to commit on your behalf is the kind of
    convenience that becomes a footgun. Drafts in `release-drafts/` keep the
    cross-repo step explicit.

## The update manifest (`update_manifest.json`)

The in-app auto-updater in `rag_gui.py` does **not** use a hardcoded file list.
Instead it fetches `update_manifest.json` from the release tag and downloads
every file the manifest lists. This is the single source of truth for "what
files make up an install."

The manifest is generated by `release.py` from the `MANIFEST_FILES` list near
the top of the script. To add a file to future auto-updates (a new module, a
renamed asset, etc.), add it to `MANIFEST_FILES` — no change to the updater is
needed. Each manifest entry carries a SHA256 and byte count so the updater can
verify integrity now or later.

```json
{
  "version": "6.0.2",
  "generated": "2026-05-19T14:30:00",
  "files": [
    { "path": "rag_gui.py", "sha256": "…", "bytes": 627614 },
    { "path": "COMPLETE_USER_GUIDE.md", "sha256": "…", "bytes": 113647 }
  ]
}
```

**Why this exists:** the original updater shipped a hardcoded 5-file list that
never included the user guide, so auto-updated clients ran new code against a
stale guide. The manifest fixes that class of bug permanently. If the manifest
is ever missing at the tag (older release, forgotten commit), the updater falls
back to a minimal hardcoded list so the update path never bricks — but the
manifest is the supported mechanism and should always be present.

`update_manifest.json` lives at the repo root and **must be committed and
pushed with every release** so it's fetchable at the tag URL
(`https://raw.githubusercontent.com/dvavro/AI-Prowler/v{version}/update_manifest.json`).

## Re-cutting a release (`--remanifest`)

Sometimes you need to correct an already-tagged release whose code changed but
whose version number stays the same — e.g. you tagged `v6.0.2`, then found bugs,
and nobody has consumed `v6.0.2` yet so bumping to `v6.0.3` would waste a number.

```cmd
release --remanifest
```

This runs the re-cut path for the **current** version (no bump): it runs the
tests, regenerates `update_manifest.json` with hashes matching your corrected
files, regenerates the release drafts, recompiles the installer, and prints the
git commands to force-move the tag.

The catch with re-cutting: editing the GitHub **release page** does NOT move the
git **tag**, and the auto-updater reads raw files at the tag, not the release
page. So you must **force-move the tag** to the new commit:

```cmd
git add -A
git commit -m "Correct v6.0.2 — auto-update fixes"
git tag -f v6.0.2
git push origin main
git push -f origin v6.0.2
```

Then on the GitHub release page: delete the old `AI-Prowler_INSTALL.exe` asset
and upload the freshly-compiled one from `Output/`, so fresh-install users also
get corrected code.

**Force-moving a tag is normally discouraged** (it rewrites published history).
It is acceptable ONLY when no one has consumed the tag yet — which is exactly
the situation a same-number re-cut implies. Confirm nobody has updated before
doing this.

**Important — what a re-cut does and does not fix:** re-cutting `v6.0.2` makes
the *files at the tag* correct, so any client already on `v6.0.2` will get the
corrected code on its NEXT auto-update. It does NOT retroactively fix clients
running an OLDER version — they still have whatever updater code shipped in
their version. Auto-update fixes only take effect from the version that contains
them, forward.

## The `suppress_when_update_available` notification flag

There are two upgrade paths a running client can offer, and showing both at once
causes a double-install trap:

  - **Green "Download Update" banner** — in-place file copy, no uninstall.
    Driven by `latest_version` in `notifications.json`. This is the correct
    path for an existing install.
  - **Yellow "Install vX" notification card** — points at the full installer
    `.exe`. The installer may require an uninstall first, after which the
    in-place updater is gone. This path is for NEW users who don't have
    AI-Prowler yet, and they reach it from the GitHub release page — not from
    inside a running copy.

To prevent a running client from showing both, add this flag to any
full-installer notification in `notifications.json` (admin repo):

```json
{
  "id": "Features_of_version_6_0_3",
  "title": "AI-Prowler™ Version 6.0.3 Now Available",
  "link_text": "Install V6.0.3",
  "link_url": "https://github.com/dvavro/AI-Prowler/releases/download/v6.0.3/AI-Prowler_INSTALL.exe",
  "suppress_when_update_available": true,
  "show_once": true
}
```

When the client detects an in-app update is available, it sets an internal flag
and `_display_notifications` skips any notification carrying
`suppress_when_update_available: true`. So existing users see only the green
in-place updater; the yellow installer card is hidden for them but still serves
new users who reach the release page directly.

## Inno Setup compiler location

The script looks for `ISCC.exe` in this order:

  1. `ISCC_EXE` environment variable (override)
  2. `C:\Program Files (x86)\Inno Setup 6\ISCC.exe`
  3. `C:\Program Files\Inno Setup 6\ISCC.exe`
  4. Anything named `ISCC` on `PATH`

If none are found, you get a clear "install Inno Setup" message and the
compile step is skipped — the rest of the release still completes.

## Flags

| Flag               | Effect                                                |
|--------------------|-------------------------------------------------------|
| `--check`          | Verify repo state, print current version, exit.        |
| `--refactor-only`  | First-time VERSION introduction. Does not bump.        |
| `--remanifest`     | Re-cut: regenerate manifest + drafts + installer for the CURRENT version, no bump. |
| `--skip-tests`     | Skip pytest (hotfix iteration only — not recommended). |
| `--skip-compile`   | Bump only; don't build installer.                      |
| `-y` / `--yes`     | Auto-confirm all prompts (CI/scripted use).            |

## Files this tool touches

| File                          | What changes                          |
|-------------------------------|---------------------------------------|
| `VERSION`                     | New version string.                   |
| `rag_gui.py`                  | (refactor only) APP_VERSION reader.   |
| `AI-Prowler-Setup.iss`        | (refactor only) `#define MyAppVersion`. |
| `COMPLETE_USER_GUIDE.md`      | `**Version X.Y.Z**` banner.           |
| `update_manifest.json`        | Generated; lists install files + SHA256. Commit + push. |
| `release-drafts/*.json`       | Generated; review then copy to admin. |
| `Output/AI-Prowler_INSTALL.exe` | Built by Inno Setup.                |

## What the tool deliberately does NOT touch

  - Anything in the admin repo (`ai-prowler-public/`, `ai-prowler-subs/`).
    Drafts only.
  - Git history (no auto-commit, no auto-tag).
  - GitHub (no auto-release, no auto-upload).
  - Beta-tester emails.

## Gitignore additions you may want

```
release-drafts/
Output/
.release-config.json
```

**Do NOT gitignore `update_manifest.json`** — it must be committed and pushed
so the auto-updater can fetch it at the tag URL. It's a tracked release
artifact, unlike the drafts and the installer output above.
